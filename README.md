Welcome to my portfolio for the **System Analysis and Design (SAD)** course.

## 📑 Overall Course Reflection

System Analysis and Design has changed how I look at software development by shifting my focus from coding to system planning and architecture. Throughout the course, I learned that building a successful system requires checking its operational, economic, and technical feasibility before any development begins. Mastering tools like Data Flow Diagrams (DFDs) and CRUD matrices taught me how to properly track data movement, manage user access levels, and map out realistic project timelines using the Critical Path Method. Ultimately, this course helped me understand how to take unstructured business problems, gather accurate requirements from stakeholders, and turn them into clear, structured blueprints that development teams can easily follow.

## 🛠️ Exercise-by-Exercise Reflection

### 🔹 [Techniques in Analysis and Design: Academic Writing]
* **Concepts Focused:** Requirements engineering, traditional vs. modern gathering methodologies, and user-centric design paradigms.
* **Implementation Details:** Analyzed how fundamental requirements gathering closes the gap between customer expectations and product delivery, transitioning from traditional data mining to modern user experience (UX) evaluation frameworks.
* **Reflection:** This assignment highlighted that a system's failure rarely results from poor coding; it mostly stems from misunderstood requirements. Learning how to properly extract user needs helps minimize project risks and ensures maximum usability right at the initialization step.

### 🔹 [Assignment 2: Project Management & Feasibility]
* **Concepts Focused:** Feasibility Assessments (Operational, Economic, Technical), Cost-Benefit Analysis (ROI, NPV, Profitability Index), and the Critical Path Method (CPM).
* **Implementation Details:** Formulated financial spreadsheets mapping the Present Value (PV) of production costs against accumulated benefits to compute a Profitability Index, alongside modeling CPM networks to isolate the project's Critical Path.
* **Reflection:** A system analyst must think like a business strategist. This exercise taught me how to mathematically justify whether a software project is worth pursuing financially and how to discover project schedule bottlenecks using critical path sequencing.

### 🔹 [Assignment 4: Data Flow Diagram Modeling]
* **Concepts Focused:** Context Diagrams, Level 0 Diagrams, and Child (Level 1) Process Decompositions.
* **Implementation Details:** Designed an end-to-end DFD suite for an order processing business (`Fruity Fresh Juice`), establishing external entities, decomposing core processes (Order Recording, Bill Calculation), and structuring child processes.
* **Reflection:** DFDs taught me how to trace data cleanly through a system. It visually enforces the rule that data cannot just appear out of nowhere—it must be accepted, processed, stored, and systematically distributed.

### 🔹 [Industry Talk: Modern Project Management Insights]
* **Concepts Focused:** Agile vs. Waterfall SDLC methodologies, enterprise project coordination, and collaborative tool stacks.
* **Implementation Details:** Engaged with Mr. Ts. Abdul Alim Abdul Mutallib (Theta Technology Telecommunication) to analyze real-world software transitions, pivot strategies, and industry communication tools like GitHub and MS Project.
* **Reflection:** Hearing from an industry lead put our classroom theory into perspective. It emphasized that project management isn't just paperwork; it is a live coordination system driven by Agile iterations, open collaboration tools, and constant communication.

## 🚀 Final Group Project Reflection: "KADA Koperasi System"

### 📄 [Lembaga Koperasi Kakitangan KADA System Architecture]
* **Concepts Covered:** Logical & Physical DFDs, Data Store Architecture, Process Decompositions, and CRUD Matrix Security mapping.
* **Core Design:** Engineered a complete digital transformation blueprint for a cooperative membership and loan application system, explicitly handling user roles (Applicants, Admins, Ahli Lembaga Koperasi) and structural entities.
* **Reflection:** Spending focused time defining architectural lines *before* diving into database creation saved countless errors. Mapping out **Logical DFDs** to show pure data flow alongside **Physical DFDs** to differentiate manual human verification from automated email and dividend routines showed me the value of professional-tier software planning.

### 📊 [Data Integrity & Access Control Validation]
* **Implementation Victory:** Successfully translated our abstract process models into a concrete data governance framework using detailed **CRUD matrices**.
* **Mechanics Validation:** Formulated strict cross-referencing permissions across distinct operational scenarios (Membership Application, Membership Approval, Loan Application, and Loan Approval) to define exactly which system role holds Create, Read, Update, or Delete authority over our system's core data stores.
* **Reflection:** Synthesizing complex data stores (Applicant Data, Membership Lists, Loan Databases) with various organizational roles forced us to think deeply about system architecture, data integrity, and security permissions. It proved how robust modeling leads directly to reliable and secure system performance under real-world multi-user loads.
